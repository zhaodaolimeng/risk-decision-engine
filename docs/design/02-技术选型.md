# 技术选型 (MVP)

## 1. 编程语言

| 选项 | 优势 | 劣势 | 选型 |
|------|------|------|------|
| Go | 高性能、并发好、部署简单、生态完善 | 泛型支持较弱 | ✅ 选定 |
| Java | 生态成熟、企业级应用多 | 内存占用大、启动慢 | - |
| Python | 开发快、AI生态好 | 性能较差、GIL限制 | - |

**选型理由**：
- 决策引擎对性能要求高，Go 的高并发特性适合
- 部署简单，单一二进制文件，便于容器化
- 标准库强大，网络编程、并发编程支持好

---

## 2. Web 框架

| 选项 | 优势 | 劣势 | 选型 |
|------|------|------|------|
| Gin | 性能好、生态完善、文档丰富 | - | ✅ 选定 |
| Echo | 性能好、API 设计优雅 | 生态略逊于 Gin | - |
| Fiber | 基于 fasthttp，性能极高 | 和标准库不兼容 | - |

**选型理由**：
- Gin 是 Go 生态最流行的 Web 框架
- 中间件丰富，文档完善
- 团队学习成本低

---

## 3. 数据库

### 3.1 关系型数据库

| 选项 | 优势 | 劣势 | 选型 |
|------|------|------|------|
| MySQL | 开源、成熟、运维经验丰富 | - | ✅ 选定 |
| PostgreSQL | 功能强大、支持 JSON | 运维经验相对少 | - |

### 3.2 缓存

| 选项 | 优势 | 劣势 | 选型 |
|------|------|------|------|
| Redis | 性能好、数据结构丰富、持久化支持 | - | ✅ 选定 |

**选型理由**：
- MySQL + Redis 是经典组合，运维成熟
- 规则、策略需要缓存加速
- Redis 可以做分布式锁、限流等

---

## 4. ORM

| 选项 | 优势 | 劣势 | 选型 |
|------|------|------|------|
| GORM | 功能完善、生态好、文档丰富 | 性能略低 | ✅ 选定 |
| sqlx | 轻量、性能好 | 需要手写 SQL | - |
| ent | 类型安全、代码生成 | 学习成本高 | - |

**选型理由**：
- GORM 是 Go 生态最流行的 ORM
- 开发效率高，适合 MVP 快速迭代
- 后续性能瓶颈可以优化

## 6. 日志

| 选项 | 优势 | 劣势 | 选型 |
|------|------|------|------|
| Zap | 高性能、结构化日志 | - | ✅ 选定 |
| Logrus | API 友好、生态好 | 性能略低 | - |

**选型理由**：
- 决策引擎高并发场景下，Zap 性能优势明显
- 结构化日志便于后续分析

---

## 7. 表达式引擎

| 选项 | 优势 | 劣势 | 选型 |
|------|------|------|------|
| expr | 高性能、类型安全、语法友好 | - | ✅ 选定 |
| goja | 完整的 JS 引擎 | 性能较低 | - |
| 自研 | 完全可控 | 开发成本高 | - |

**选型理由**：
- expr 专为表达式求值设计，性能优秀
- 语法类似 Go，学习成本低
- 支持函数、变量，满足规则需求
